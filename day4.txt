Day 4 — Classification (Part 1) Assessment Report
Topic: Binary Classification Mastery (Heart Disease Prediction)

1) Objective
Today’s goal was to understand and practice binary classification using a real healthcare-style dataset (Heart Disease UCI-like). The focus was not only “training a model”, but learning how to evaluate it correctly—especially when data can be imbalanced and accuracy can be misleading.

2) Dataset & Target Definition
- Total patients (rows): 920
- Initial issue: multiple columns had missing values, with some columns missing more than half their data.

Target selection:
- The dataset column `num` represents the diagnosis outcome.
- For binary classification we created a new target:
  - target = 0  if num == 0  (No disease)
  - target = 1  if num != 0  (Disease present)
This converted a multi-level diagnosis column into a clean YES/NO label suitable for binary classification.

We also removed the `id` column from features because it is only an identifier and does not carry medical meaning. Keeping `id` could mislead the model.

3) Handling Missing Data (Healthcare-Safe Baseline)
We used a “safe baseline” approach:
- Drop extremely incomplete columns:
  - `ca` (~66% missing)
  - `thal` (~53% missing)
Reason: Filling these would require guessing values for most patients, which can distort a medical model.

For the remaining columns:
- Categorical columns (e.g., slope, restecg, cp, sex, origin):
  - Filled missing values with "Unknown"
  Reason: This avoids guessing medical categories and preserves rows.

- Boolean-like columns (fbs, exang):
  - Also treated as categorical and filled with "Unknown"
  Reason: Missing does not mean True or False; we keep an honest “unknown” label.

- Numeric columns (trestbps, chol, thalach/thalch, oldpeak, age):
  - Filled missing values with median
  Reason: Median is more robust than mean when medical data contains outliers.

Important bug encountered and solved:
- After filling boolean columns with "Unknown", some columns contained both booleans (True/False) and strings ("Unknown").
- This caused OneHotEncoder to fail.
Fix:
- Convert those mixed-type columns to strings so all categories are uniform (e.g., "True", "False", "Unknown").

4) Why We Needed a Preprocessing Block
Even after missing values were handled, the dataset still contained categories (text labels).
Models like Logistic Regression and Decision Tree require numeric inputs.

So we built a preprocessing pipeline to:
- Scale numeric columns (optional but helpful for Logistic Regression stability).
- One-hot encode categorical columns so categories become numeric columns.
This allowed clean, repeatable training without manual conversions.

5) Evaluation Setup (Correct Metrics for Classification)
We created a reusable evaluation function for binary classification that prints:
- Confusion Matrix (TP, TN, FP, FN)
- Classification report (precision, recall, F1)
- ROC-AUC and PR-AUC (how well the model ranks risk)
- Accuracy (included, but not trusted alone)

Key mindset learned:
- Accuracy can be misleading, especially in imbalanced datasets.
- In healthcare, Recall (catching sick patients) is often more important than Accuracy.

6) Models Trained and Compared

A) Logistic Regression (Baseline Classifier)
Results:
- Confusion Matrix: [[64, 18], [15, 87]]
- Accuracy: ~0.821
- Class 1 (Disease): Precision ~0.829, Recall ~0.853, F1 ~0.841
- ROC-AUC: ~0.920
- PR-AUC: ~0.922

Interpretation:
- Strong overall baseline.
- Very good at ranking risk (high ROC-AUC and PR-AUC).
- Missed 15 disease cases on the test set.

B) Decision Tree Classifier (Initial + Tuned)
Initial tree performed worse:
- Missed many disease cases (high false negatives).
We learned that a tree can overfit or become unstable if it becomes too complex.

We tuned max_depth and found the best-performing simple tree:

Best Decision Tree (max_depth=2):
- Confusion Matrix: [[60, 22], [12, 90]]
- Accuracy: ~0.815
- Class 1 (Disease): Precision ~0.804, Recall ~0.882, F1 ~0.841
- ROC-AUC: ~0.857
- PR-AUC: ~0.853

Interpretation:
- Caught slightly more disease cases than Logistic Regression (FN=12 vs FN=15).
- Produced more false alarms (FP=22 vs FP=18).
- Worse overall ranking power than Logistic Regression (lower AUC scores).

7) Final Conclusion (Model Comparison)
- Logistic Regression performed best overall, especially on ROC-AUC and PR-AUC, meaning it ranks patient risk very well.
- Decision Tree (depth=2) improved disease recall slightly (caught more sick patients) but increased false alarms and had weaker AUCs.

Practical takeaway:
- If the goal is the strongest general-purpose classifier and best risk ranking → Logistic Regression is the better choice.
- If the main goal is maximizing disease recall (missing fewer sick patients) and extra false alarms are acceptable → a shallow Decision Tree (depth=2) can be considered.

8) Model Saving
We rebuilt the best Decision Tree (max_depth=2), trained it, and saved it using joblib:
- File saved: best_decision_tree_depth2.joblib
This ensures the trained pipeline (including preprocessing + model) can be reused without retraining.

9) What We Learned (Core Skills Gained)
- How to define a correct binary target from a multi-level label.
- How to handle missing data responsibly in healthcare contexts.
- Why categorical values must be encoded for ML models.
- How to evaluate classifiers properly using confusion matrix, precision, recall, F1, ROC-AUC, and PR-AUC.
- How accuracy can lie, and why recall and PR-AUC are crucial in medical-style datasets.
- How tuning a Decision Tree (depth control) can significantly change performance.
- How to save a trained model pipeline for later use.
