Day 2 Report — California Housing Price Prediction (Regression)

Project: California Housing Price Prediction
Goal: Predict median_house_value using regression models and compare improvements.

1) What I worked on today
- Loaded and explored the California Housing dataset.
- Cleaned the dataset (handled missing values).
- Performed feature engineering to create more meaningful inputs.
- Split the data into training and testing sets to evaluate fairly.
- Built a clean preprocessing workflow to prepare the data for models.
- Trained and evaluated multiple regression models:
  1) Linear Regression (baseline)
  2) Ridge Regression (regularized linear model)
  3) Lasso Regression (regularized model that can ignore weak features)
  4) Polynomial Regression (degree=2) + Ridge (captures non-linear patterns)
- Saved the trained model to a file for later reuse.

2) Understanding the dataset (what the data looks like)
- The dataset contains housing-related information such as:
  - location (longitude, latitude)
  - housing age
  - total rooms, bedrooms, population, households
  - median income
  - ocean_proximity (categorical label like INLAND / NEAR BAY etc.)
- Target (what we predict): median_house_value

3) Missing values (what I found and what I did)
- The column total_bedrooms had a small number of missing values.
- Because it was ~1% of the data, I chose to drop those rows.
- After dropping, I confirmed there were no remaining missing values.

4) Feature Engineering (why I created new columns)
I learned that totals like total_rooms or population are not always directly useful by themselves because:
- larger areas naturally have larger totals
- totals don’t reflect “average house characteristics”

So I created ratio-based features to describe the average situation per household:
- rooms_per_household = total_rooms / households
  Meaning: average number of rooms per household (roughly home size indicator)
- bedrooms_per_room = total_bedrooms / total_rooms
  Meaning: what fraction of rooms are bedrooms (layout / density signal)
- population_per_household = population / households
  Meaning: average people per household (crowding indicator)

I also learned safe coding practice:
- avoid division-by-zero by replacing 0 denominators with NaN, then dropping.

5) Correlation check (what I learned from correlation)
- I used correlation with the target to see which features have stronger relationships with house price.
- median_income was the strongest positive relationship with median_house_value.
- bedrooms_per_room had a noticeable negative relationship, meaning areas with a higher bedroom fraction often had lower prices (likely smaller/tighter housing layouts).

Important note I learned:
- Correlation helps me understand relationships, but it does not automatically mean a feature is “the only important one”.
- A model can still benefit from combinations of features (especially with polynomial features).

6) Train/Test split (why we split the data)
- Training set: used to learn patterns.
- Test set: used as the final “exam” to check performance on unseen data.
- This prevents me from fooling myself by testing on data the model already saw.

7) Handling numeric vs categorical columns (why this matters)
- Numeric features (numbers) can be scaled and used directly.
- Categorical features (text like ocean_proximity) must be converted into numbers.
- I used one-hot encoding to convert ocean_proximity into 0/1 columns.

8) Preprocessing pipeline (what it means in simple terms)
I built an automated “data preparation machine” so every model receives clean, consistent inputs:

For numeric columns:
- fill missing values using median
- scale values using StandardScaler (keeps features on comparable scale)

For categorical columns:
- fill missing values using the most common category
- one-hot encode the category

Why this is important:
- models work better when numeric scales are consistent
- encoding makes categorical data usable
- pipelines prevent mistakes (same cleaning is applied to both train and test)

9) Models I trained and what the results mean
I evaluated models using:
- RMSE (Root Mean Squared Error): penalizes large mistakes more
- MAE (Mean Absolute Error): average absolute mistake

Baseline (Linear Regression):
- Linear -> RMSE: 69042.80 | MAE: 49774.51
Meaning:
- This is my starting benchmark.
- Any “better” model should beat this (lower RMSE/MAE).

Ridge Regression (regularized linear regression):
- Best Ridge alpha: 0.1
- Ridge(best) -> RMSE: 69036.69 | MAE: 49772.00
Meaning:
- Ridge performed almost the same as Linear.
- This suggests the plain linear model was already stable and not strongly overfitting in this feature setup.

Lasso Regression (regularized + can zero out weak features):
- Best Lasso alpha: 1
- Lasso(best) -> RMSE: 69036.81 | MAE: 49771.94
Meaning:
- Lasso also performed almost the same as Linear/Ridge.
- It did not significantly improve performance here.

Polynomial Regression (degree=2) + Ridge:
- Poly(deg=2)+Ridge(alpha=10) -> RMSE: 65212.81 | MAE: 46612.52
Meaning:
- This was a major improvement.
- It shows the data likely contains non-linear relationships (curved patterns / interactions between features).
- Adding polynomial features helped the model capture those patterns.

10) Key conclusions from today
- Linear, Ridge, and Lasso all performed similarly (around RMSE ~ 69k).
- Polynomial degree=2 + Ridge gave the best improvement (RMSE ~ 65k).
- This suggests:
  - the relationship between housing features and price is not purely straight-line
  - engineered ratio features and polynomial terms can improve prediction
